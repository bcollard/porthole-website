Why

The friction between a developer and a misbehaving pod, removed.

Lens and k9s are great if every developer has cluster credentials and a kubeconfig. Most don't. porthole hands them a URL, an OIDC login, and a terminal — backed by ephemeral containers so the target pod stays exactly as deployed.

01

Browser-first

A single page, an embedded xterm.js terminal, a single websocket. Stdin/stdout flow as binary frames; resize travels as a JSON control frame. No kubectl on the developer's laptop. No SSH bastion.

02

Ephemeral, not invasive

porthole patches the pod's ephemeralcontainers subresource — the workload container is untouched and stays in place. When the websocket closes, the debug container can be evicted by the next pod restart.

03

Any debug image

Default is nicolaka/netshoot. Pass busybox, alpine, postgres for a psql client, or your own forensic image. Pulls go through your registry's normal auth.

04

OIDC at the edge

Front porthole with an Envoy Gateway SecurityPolicy (or your gateway of choice). porthole only trusts JWTs it can verify against the JWKS endpoint — no header forgery, no shared secrets.

05

OPA-decided authZ

An OPA sidecar runs the Rego your platform team writes. Group memberships from the JWT, namespace labels resolved live, time-windows, glob matchers — everything is data, not code.

06

One JSON line per decision

Structured slog output, one line per inject and per attach deny. SIEM-friendly out of the box: user, namespace, pod, image, outcome, reason.

Under the hood

One websocket. Three byte paths. Zero magic.

Browser → gateway → porthole → kube-apiserver → kubelet → PTY. porthole multiplexes three flows over the single websocket: stdin and stdout as binary frames, resize as a JSON text frame. The k8s remotecommand executor stitches the upstream half.

Browser ↔ porthole

Binary frames carry stdin/stdout. {"type":"resize","cols":N,"rows":M} as a text frame keeps the PTY size in sync with the browser. xterm.js handles the rendering.

porthole ↔ kube-apiserver

Standard POST /api/v1/namespaces/<ns>/pods/<p>/attach via the client-go remotecommand executor — SPDY with WebSocket fallback. A TerminalSizeQueue propagates resize events to the kubelet.

kubelet ↔ container PTY

Vanilla CRI streaming. Resize events become TIOCSWINSZ ioctls + SIGWINCH. Source-IP and TLS terminate at the gateway — porthole runs cleartext inside the cluster.

Identity & authorization

Identity at the edge. Decisions next door.

porthole separates who from what's allowed. Your gateway proves who. OPA decides what. Both arrows are independent so you can swap either side.

Authentication

The gateway runs the OIDC handshake (against Keycloak, Auth0, Google, whatever) and forwards the validated request upstream with the token in a header. porthole reads X-ID-Token first, falls back to Authorization: Bearer …, validates against the configured JWKS endpoint, extracts sub, email, groups.

# SecurityPolicy on Envoy Gateway
spec:
  oidc:
    provider:
      issuer: "https://idp.example.com/realms/eng"
    clientID: "porthole"
    clientSecret:
      name: porthole-oidc-secret
    scopes: [openid, profile, email]
    forwardAccessToken: true

Authorization

An OPA sidecar runs alongside porthole. Each list_pods, inject_ec, attach_ec is sent as input — including the live namespace labels — and OPA returns {allow, reason}. The reason flows back into the audit log.

# Rego — group × namespace_glob × labels × time
decision := {"allow": true, "reason": reason} if {
  some binding in data.policy.bindings
  binding.group in input.user.groups
  matches_namespace(binding, input.request.namespace)
  input.request.action in data.policy.roles[binding.role]
  not violates_time_window(binding)
  reason := "matched"
}
Policy is data. Roles, bindings, and namespace-label requirements live in a JSON file mounted as a ConfigMap. opa eval against the file gives the platform team a regression suite that runs anywhere.

Audit

One JSON line per security-relevant action.

Every inject — success, error, or deny — emits an slog JSON line on stdout. Same for every attach that the policy refuses. Picks up cleanly in Loki, Splunk, or any structured-log pipeline.

Self-host it

One Go binary, one ConfigMap, one gateway.

Build with ko, apply the manifests, set OIDC + OPA env vars. The repo ships a working Makefile, Envoy Gateway manifests, and a Keycloak bootstrap script for trying it end-to-end. 

$ git clone https://github.com/bcollard/porthole
$ cd porthole
$ make e2e             # cluster + envoy + opa + keycloak

Prereqs. A Kubernetes cluster (kind works), kubectl, helm, ko, an OIDC provider, and the opa CLI for local policy testing.

  • Envoy Gateway 1.6+ for the supplied SecurityPolicy.
  • OPA 0.69+ runs as a sidecar in the porthole pod.
  • An OIDC client registered with your IdP and a redirect URL pointing at the gateway.
  • RBAC: porthole's SA needs patch on pods/ephemeralcontainers and create on pods/attach.

Then a typical request looks like this

# Developer opens https://porthole.example.com/ui/, signs in via SSO,
# picks namespace 'payments', then pod 'api-7d4b9c5f-mznpr'.

# Browser → POST /debug/inject
{"namespace":"payments","pod":"api-...","image":"nicolaka/netshoot"}

# porthole → OPA
{"input":{"user":{"groups":["team-payments"]},
          "request":{"action":"inject_ec","namespace":"payments"},
          "namespace_labels":{"tier":"production"}}}
# OPA → porthole
{"result":{"allow":true,"reason":"matched: group=team-payments role=debugger ns=payments"}}

# porthole → kube-apiserver (PATCH ephemeralcontainers) → kubelet starts the EC
# Browser opens WS /term/payments/api-.../porthole-a2045e61 → attached.

FAQ

Common questions.

How is this different from kubectl debug or Lens?

kubectl debug requires every developer to have a working kubeconfig and the right RBAC. Lens is great but assumes the same. porthole removes the kubeconfig from the developer's laptop entirely — the cluster sees a single SA (porthole's), and a corporate OIDC identity gets translated into namespace-scoped permission via OPA. The footprint on developers is just a URL.

Why ephemeral containers?

They give you kubectl exec ergonomics without modifying the workload container. Different image (your forensics toolkit, psql, …), same network namespace, same volumes as the target. The pod's deployment YAML stays untouched.

Why an OPA sidecar instead of in-process rules?

Because policies grow. The Rego file in the repo demonstrates: groups from the JWT, namespace label match, glob namespace matching, business-hours constraints — combined in arbitrary ways without recompiling porthole. Sidecar gives you hot reload on a ConfigMap update.

Does porthole need its own database?

No. All state lives in Kubernetes (the ephemeral container, the kube events) or in the JWT (identity). porthole itself is stateless — you can run multiple replicas behind the gateway.

What does it log?

One slog JSON line per inject and per attach-deny: user, source_ip, namespace, pod, image, outcome, reason, duration_ms. Successful attaches show up in gin's access log; the byte stream is never logged.

Can I use a gateway other than Envoy Gateway?

Yes. Anything that does OIDC and forwards a JWT in a header upstream works — Kong, Traefik EE, Apache APISIX, Cloudflare Access, an oauth2-proxy in front. porthole only cares that X-ID-Token or Authorization: Bearer … contains a JWT that validates against the JWKS URL you set.

How does resize work over the websocket?

The websocket is the only channel. Stdin and stdout travel as binary frames; the browser sends a JSON text frame for resize: {"type":"resize","cols":120,"rows":30}. porthole pushes that into a remotecommand.TerminalSizeQueue; the kubelet turns it into TIOCSWINSZ on the PTY.

Is it open source?

Yes. Source on GitHub. Issues and PRs welcome.
